## What this is

A hand-runnable verification of the P3 extraction: the `preservelib` code lifted into the standalone `dazzle-preservelib` library (L3 of the DazzleLib stack), delegating OS mechanics to `dazzle-filekit` (L1) and the `.dazzlelink` record to `dazzle-linklib` (L2), with the `preserve` CLI rewired to consume it. Run by a maintainer before trusting the 0.8.x line in production.

## High-Value Verification (~5 minutes)

### HV.1 -- Clean-venv install from PyPI (base + extra)

Validates: the published artifact is whole and self-resolving.

- [ ] **Steps** (cross-shell safe, uses `python`/`pip` only):
  ```
  python -m venv /tmp/dpl-check          # POSIX;  py -m venv %TEMP%\dpl-check on Windows
  <venv>/python -m pip install dazzle-preservelib
  cd /tmp                                 # NEUTRAL cwd (not a repo checkout)
  <venv>/python -c "import dazzle_preservelib as d; print(d.__file__, d.__version__)"
  ```
- [ ] **Expected:** import path is under `site-packages` (NOT a source checkout); version starts `0.8`.
- [ ] **Why this matters:** if this fails, the wheel is broken regardless of what the test suite says.

### HV.2 -- The `[dazzlelink]` bridge round-trips

- [ ] **Steps:** `pip install "dazzle-preservelib[dazzlelink]"`, then from a neutral cwd:
  ```
  python -c "from dazzle_preservelib import dazzlelink as b; print(b.is_available())"
  ```
- [ ] **Expected:** `True`. (Without the extra, `is_available()` is `False` and the bridge raises a clear named error, never a silent fallback.)

### HV.3 -- preserve CLI drives a real COPY -> VERIFY against the lib

- [ ] **Steps:** in a dir with a couple of files, run the real CLI: `preserve COPY file1 file2 --dst <out> --hash SHA256` then `preserve VERIFY --src . --dst <out>`.
- [ ] **Expected:** copy reports success; verify reports all-match. A `preserve_manifest.json` appears in `<out>`.
- [ ] **Why this matters:** this is the end-to-end consumer proof -- the lifted library doing the actual work through the CLI.

### HV.4 -- Sequential manifest numbering + migration (next_manifest_path)

- [ ] **Steps:** run a SECOND `preserve COPY ... --dst <out>` into the same destination.
- [ ] **Expected:** the console prints `Migrating preserve_manifest.json to preserve_manifest_001.json`; the dir now holds `preserve_manifest_001.json` + `preserve_manifest_002.json`. A third op yields `_003`.
- [ ] **Why this matters:** the WRITE-side lifecycle (`next_manifest_path`) pulled down from the CLI must migrate exactly once and never reuse numbers.

### HV.5 -- Hard-link-on-directory error names the reason

- [ ] **Steps:** attempt to hard-link a directory (e.g. via a MOVE with `--link-handling` on a dir, or directly `python -c "from dazzle_preservelib.links import create_link, LINK_TYPE_HARD; print(create_link('lnk','somedir',link_type=LINK_TYPE_HARD))"`).
- [ ] **Expected:** failure whose message contains "files" ("Hard links can only be created for files, not directories") -- NOT a generic "creation failed". (This was the conservation regression the rewire caught.)

### HV.6 -- InsufficientSpaceError reports source AND destination

- [ ] **Steps:**
  ```
  python -c "from dazzle_preservelib.operations import InsufficientSpaceError as E; e=E(100,50,'D:/dest',source='C:/src'); print(e.source, '|', e.destination, '|', str(e))"
  ```
- [ ] **Expected:** `C:/src | D:/dest | ...copying from 'C:/src'`. Constructing with only 3 args still works and `.destination` is intact (CLI back-compat).

## Section 1: Manifest lifecycle (real filesystem)

### 1.1 `.preserve/`-aware discovery
- [ ] Put a `preserve_manifest.json` only inside a `.preserve/` subdir; confirm `find_available_manifests(dir, include_preserve_subdir=True)` finds it, and the default (no flag) does NOT.

### 1.2 describe_manifest on a real manifest
- [ ] After a COPY, `python -c "from dazzle_preservelib.manifest import describe_manifest; print(describe_manifest('<out>/preserve_manifest.json'))"` -> dict with number 0, file_count matching, a manifest_id. On a corrupt/missing file -> `None`.

### 1.3 scan-only does not migrate
- [ ] With an existing single manifest, a `--scan-only` style read must NOT rename it to `_001` (predict-only). Verify the file is untouched on disk.

## Section 2: preserve consumer integration

### 2.1 Full round-trip with a symlink + junction (Windows)
- [ ] COPY a tree containing a symlink and (Windows) a junction -> VERIFY -> MOVE -> RESTORE; results byte-identical to source. Watch for link-policy decisions.

### 2.2 dazzlelink mode through the CLI
- [ ] Run a COPY with `--dazzlelink`; confirm `.dazzlelink` files are written and restore resolves the original path. Requires `dazzle-preservelib[dazzlelink]`.

### 2.3 No `preservelib` left behind
- [ ] `pip show dazzle-preserve` (once published) pulls `dazzle-preservelib`; the installed `preserve` package contains NO bundled `preservelib/`.

## What the automated tests DON'T cover

- **Real PyPI resolution** of the full dep chain on a fresh machine (mocks/editable installs hide missing-wheel-file bugs) -- HV.1 covers it.
- **Windows junction/symlink privilege** behavior (junctions need no elevation; symlinks do) on a non-Developer-Mode box.
- **Console rendering** of the "Migrating ..." message and verify tables (codepage 437/1252, width).
- **Cross-platform** link behavior on Linux/macOS (POSIX symlinks/hardlinks via filekit).
- **Real disk-full** triggering InsufficientSpaceError with a populated `source` through an actual copy.
- The **`preserve` PyPI publish** itself (not yet released; HV.3 uses the local/installed CLI).
